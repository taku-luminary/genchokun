# 開発方針

ユーザーは React / Next.js の初学者です。
コードをすべて理解しながら開発したいため、最終的なコード反映はユーザー自身が行います。
ファイルを直接編集せず、コード提案と説明に徹してください。

# 出力ルール

- なるべく簡単で分かりやすい実装を提案する
- 複雑な抽象化や過剰実装を避ける
- まずは MVP 完成を優先する
- 実装前に、目的・変更対象ファイル・変更内容を簡潔に説明する
- 変更ごとに、変更後コードと理由を示す
- 一度に大きく変更しすぎず、段階的に進める
- ユーザーがコピペしやすい形で提示する
- 必要なら変更前後の差分が分かるようにする

# 要件書の扱い

`docs/requirements.md` は長文の要件書です。

- 最初の計画作成時のみ参照してよい
- 参照後は会話内の要約を前提として進める
- 以後は原則として再参照しない
- 要件の不明点・矛盾・重大な設計判断がある場合のみ最小限で再参照してよい
- ユーザーが明示的に指示した場合のみ再参照してよい

# セキュリティルール
- `.env`, `.env.local`, `.env.*.local`, `.env.production`, `.env.development` などの環境変数ファイルは読まない
- `secrets/`, `credentials/`, `keys/` など秘密情報を含む可能性のあるディレクトリやファイルは読まない
- APIキー、アクセストークン、DB接続文字列、秘密鍵、認証情報などの値を取得・表示・推測しない
- 環境変数が必要な場合は、値そのものではなく「変数名」だけを確認対象とする
- 設定例を示すときは、必ずダミー値やプレースホルダーを使う
- ユーザーに対しても、秘密情報そのものを貼り付けさせない
- 秘密情報を含むファイルに対して `cat`, `grep`, `head`, `tail`, `sed`, `awk` など内容表示につながる操作を提案・実行しない
- 秘密情報を含む可能性があるファイルの一覧確認・参照・編集・要約は行わない
- セキュリティに関わる確認が必要な場合は、ユーザーに秘密部分を伏せたうえで共有してもらう前提で進める
- コード例・説明・ログ例では、実在の秘密情報を絶対に出力しない


# 進め方

- まず現在のファイル構成と関連ファイルを確認する
- 要件を踏まえて、今やるべき最小の1ステップを選ぶ
- そのステップに必要な変更だけ提案する
- ユーザーの学習を優先し、説明は初学者向けに行う
