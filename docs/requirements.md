現調くん 要件定義（MVP前提）

■1. プロジェクト概要
1-1. 作りたいもの

現地調査をマッチングするWebアプリケーションを作成する。

1-2. マネタイズ方法

サブスクリプションモデル（月額課金）とする。
決済および請求管理は Stripe を利用する想定。

1-3. 開発方針
私は React / Next.js の初学者である
まずは MVP を作り、最低限マネタイズできるところまで進める
より良い案があれば微修正してよいが、過剰実装はしない

■2. ターゲットユーザー
このサービスの主なターゲットは以下の2者である。

2-1. 販売店
現地調査および、その後の工事を依頼したい事業者

2-2. 工事店
現地調査をフックに営業し、その後の工事受注につなげたい事業者

2-3. 補足
1人の登録ユーザーが、販売店と工事店の両方の役割を持つことがある。

■3. サービスの目的
販売店による案件投稿を促進する
工事店による募集投稿（行ける日程の掲載）を促進する
販売店と工事店のマッチングを成立させる

■4. 用語定義
4-1. 案件

販売店が投稿する「現地調査を依頼したい募集」

4-2. 依頼待ち

工事店が投稿する「現地調査に行ける日程・対応可能枠」

4-3. 応募
工事店が販売店の案件に応募すること
販売店が工事店の依頼待ちに応募すること
4-4. マッチング

販売店と工事店の成立状態


■5. 主要機能
5-1. 販売店としてできること
現地調査の募集案件を投稿できる
投稿した案件を削除できる
自分の案件に応募してきた工事店を確認できる
応募してきた工事店の中から、マッチする工事店を1つ選択できる
工事店が掲載している依頼待ちに応募できる
依頼待ちへの応募は、応募した時点で即マッチングとする
5-2. 工事店としてできること
現地調査に行ける日程を投稿できる
投稿した依頼待ちを削除できる
販売店が掲載している案件へ応募できる
応募をキャンセルできる
5-3. マイページでできること

マイページでは以下を確認できる。

自分が販売店として投稿した案件
自分が工事店として投稿した依頼待ち
自分が応募した案件
自分が応募した依頼待ち
自分の投稿に来た応募
マッチング情報

■6. 一覧・表示の考え方
6-1. マイページでの表示軸
軸1: ユーザーの役割
販売店として
工事店として
軸2: 案件との関係性
自分が掲載したもの
自分が応募したもの
6-2. トップ画面での表示軸
軸1: ユーザーの役割
販売店として閲覧
工事店として閲覧
軸2: 案件との関係性
全体案件（自分および他人が掲載したもの）

■7. 重要な仕様
7-1. デバイス想定

Webアプリケーションとして作るが、主な閲覧環境は iPhone を想定する。

7-2. マッチング仕様
販売店が出した案件に対して、成立するマッチングは1件のみ
販売店の案件には、複数の工事店が応募できる
販売店が工事店の依頼待ちに応募した場合は即マッチングとする
7-3. 支払い仕様

支払いおよび請求は Stripe を利用する。

■8. 権限
8-1. 非会員
トップページの閲覧
案件一覧の閲覧
依頼待ち一覧の閲覧
案件詳細の閲覧
依頼待ち詳細の閲覧
会社詳細の閲覧
問い合わせ

補足:投稿や応募をしようとした場合は、下院登録が必要な旨表示し、会員登録またはログイン導線に遷移させる

8-2. 無料会員
非会員ができること全部
マイページ利用
自社情報の最低限登録（簡易表示のみ）
販売店として案件投稿（月3件まで）
販売店として依頼待ちへ応募（月3件まで）
工事店として依頼待ち投稿（月3件まで）
工事店として案件へ応募（月3件まで）
自分の投稿の編集・削除
自分の応募一覧の確認
来た応募の確認
プラン変更
8-3. 有料会員
無料会員の全機能
案件投稿数 無制限
依頼待ち投稿数 無制限
応募数 無制限
企業プロフィールの充実表示
応募者情報の詳細閲覧
マッチ相手情報の詳細閲覧
契約期間中の継続利用
請求情報の確認
プラン変更
解約
8-4. 管理者
全ユーザーの閲覧・停止
全案件、全依頼待ち、全応募、全マッチの閲覧
不適切投稿の削除
ユーザーの強制停止 / isActive 管理
課金プランの作成・変更・停止
問い合わせ対応
障害対応
通報対応
ユーザー代理での状態修正
company_interview_articles の掲載・修正・削除
サブスク状態確認
不整合データ修正
